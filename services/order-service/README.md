# Order Service

Python 3.12 / FastAPI REST API for order management with cross-service stock validation, event-driven inventory updates, and Redis caching.

## Tech Stack

| Technology | Purpose |
|---|---|
| Python 3.12 | Language runtime |
| FastAPI | Async web framework |
| SQLAlchemy (async) | Database ORM with asyncpg driver |
| PostgreSQL | Relational database (Cloud SQL) |
| Redis | Cache-aside for product availability |
| Google Pub/Sub | Async event messaging (order events) |
| OpenTelemetry | Distributed tracing |
| Alembic | Database migrations |
| uv | Package and project management |

## API Endpoints

| Method | Path | Description |
|---|---|---|
| POST | `/api/orders` | Create a new order (validates stock via Product Service) |
| GET | `/api/orders/{id}` | Get order by ID |
| GET | `/api/orders?email=` | List orders by customer email |
| GET | `/api/orders/{id}/items` | Get order line items |
| GET | `/api/availability/{product_id}` | Check product availability (Redis cached) |
| GET | `/ready` | Readiness probe (database health check) |

## Running Locally

Requires PostgreSQL and Redis (or use Docker Compose from the repository root).

```bash
uv run uvicorn src.order_service.main:app --reload
```

The service starts on port 8000 by default.

## Running Tests

Tests use Testcontainers for PostgreSQL. Docker must be running.

```bash
uv run pytest
```

## Building Docker Image

```bash
docker build -t order-service .
```

The Dockerfile uses a two-phase `uv sync` strategy: dependencies first (cached layer), then project source.

## Configuration

Key configuration properties (set via environment variables):

| Variable | Description | Default |
|---|---|---|
| `DATABASE_URL` | Async PostgreSQL connection string | - |
| `REDIS_URL` | Redis connection URL | `redis://redis:6379/0` |
| `PRODUCT_SERVICE_URL` | Product Service base URL | `http://product-service:8080` |
| `GCP_PROJECT_ID` | GCP project for Pub/Sub | - |
| `OTEL_SERVICE_NAME` | OpenTelemetry service identifier | `order-service` |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | OTLP collector endpoint | - |
| `LOG_FORMAT` | Log format: `json` or `text` | `text` |

## Key Design Decisions

- **Cache-aside pattern** with best-effort Redis writes; Redis failure falls back to Product Service transparently
- **Pub/Sub publisher** raises on failure; OrderService catches to keep order PENDING (graceful degradation)
- **Async SQLAlchemy** with asyncpg for non-blocking database operations
- **Alembic migrations** managed manually (not autogenerated) for explicit schema control
- **Readiness endpoint** `/ready` performs `SELECT 1` database health check for Kubernetes probes
